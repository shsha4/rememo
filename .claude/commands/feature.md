# /feature — rememo 개발 파이프라인 오케스트레이션

요청 작업: **$ARGUMENTS**

아래 8단계를 **순서대로** 진행한다. 각 단계 완료 시 결과를 짧게 보고하고 다음 단계로 넘어간다.
`CLAUDE.md`의 아키텍처·컨벤션·Definition of Done을 기준으로 삼는다.

## 1. 프로젝트 분석
- 이 작업이 건드리는 영역을 파악한다: core / main(service·ipc) / renderer(page·component·store).
- 관련 기존 파일을 읽고 데이터 흐름을 확인한다. (CLAUDE.md §1)

## 2. 코드 컨벤션 확인
- CLAUDE.md §2와 유사 기존 코드에서 따라야 할 패턴을 확정한다.
- **진실의 원천 규칙**: 도메인/파서는 `packages/core`에서만. desktop 복붙 금지.

## 3~4. 요구사항 분석 · 플랜 작성 · 플랜 리뷰
- `plan` 스킬을 사용해 플랜을 작성하고 셀프 리뷰한다.
- 완성 플랜을 **사용자에게 제시하고 착수 동의를 받는다.** (과설계/누락 점검)

## 5. 코드 작성
- 합의된 플랜대로, CLAUDE.md §2 컨벤션을 지켜 구현한다.
- 뮤테이션이면 indexer 재인덱싱 트리거를 함께 처리한다.

## 6. 테스트 작성
- `test-writer` 스킬로 단위 테스트를 작성한다(core 순수 로직 우선, 한글/조사/경계 케이스 포함).
- `npm test` 초록불 확인.

## 7. 코드 리뷰
- `code-reviewer` 서브에이전트를 **정확성/보안·경계/성능 관점으로 병렬 실행**하거나 `/code-review`를 사용한다.
- 지적 사항을 반영하고 재검증한다.

## 8. Definition of Done 게이트 + 커밋 & 푸시 (+ PR)
- 게이트 실행: `npm run type-check` · `npm run lint`(에러 0) · `npm run format:check` · `npm test` · `npm run build` 모두 통과.
- UI/동작 변경이면 `npm run dev` 또는 `run` 스킬로 실제 앱에서 확인.
- **커밋 전 CLAUDE.md 최신화(필수)**: 아키텍처/컨벤션/검증명령/파이프라인이 바뀌었으면 `CLAUDE.md`를 **같은 커밋에** 함께 수정한다. (CLAUDE.md §커밋/PR 규칙 참고)
- **커밋/푸시는 사용자가 요청할 때만.** `main`이 아닌 새 브랜치에서 작업한다.
- **PR 생성 여부는 사용자에게 질의**한 뒤 결정한다(선택 사항).

---
각 단계에서 막히거나 결정이 필요하면 진행을 멈추고 사용자에게 확인한다.

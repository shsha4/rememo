# plan — 요구사항 분석 & 구현 플랜 (파이프라인 3~4단계)

코드를 쓰기 **전에** 무엇을/왜/어디에 만들지 확정한다. 목표는 **팀 누구나 같은 결론**에 도달할 수 있는 플랜.

## 절차

### 1. 요구사항 분석
- 사용자 요청을 한 문장 목표로 요약한다.
- **불명확하면 추측하지 말고 질문**한다(범위, 엣지케이스, UX 기대).
- 영향 범위를 분류: core(도메인/파서) / main(service·ipc) / renderer(UI·store) 중 어디를 건드리는가.

### 2. 코드베이스 조사
- `CLAUDE.md` §1(아키텍처)·§2(컨벤션)을 먼저 읽는다.
- 유사한 기존 구현을 찾아 패턴을 재사용한다(새 패턴 도입 최소화).
- **진실의 원천 규칙**: 도메인/파서 변경은 `packages/core`에서 한다. desktop에 복붙 금지.
- 데이터 흐름을 따라간다: renderer → preload → ipc(`도메인:동작`) → service → 파일/DB → indexer 재인덱싱.

### 3. 플랜 작성 (아래 형식)
```
## 목표
<한 문장>

## 영향 범위
- core: <변경/없음>
- main(service/ipc): <변경/없음>
- renderer(page/component/store): <변경/없음>

## 변경 파일 (순서대로)
1. packages/core/... — <무엇을>
2. apps/desktop/src/main/... — <무엇을>
3. apps/desktop/src/renderer/... — <무엇을>

## 데이터 흐름 / IPC
- 새 채널: `도메인:동작` (필요 시)
- 재인덱싱 트리거 필요 여부: <예/아니오>

## 테스트 계획
- core 순수 로직 단위 테스트: <케이스>
- 엣지케이스(한글/조사/경계): <케이스>

## 리스크 / 미결정
- <있으면 나열, 없으면 "없음">
```

### 4. 셀프 리뷰 (플랜 리뷰 = 파이프라인 4단계)
플랜을 확정하기 전에 스스로 점검한다:
- [ ] **과설계 아닌가?** 요청 범위를 넘는 추상화/일반화를 넣지 않았는가.
- [ ] 기존 컨벤션·패턴을 따르는가(새 패턴을 최소화했는가).
- [ ] core 진실의 원천 규칙을 지키는가(desktop 복붙 없음).
- [ ] 뮤테이션이면 인덱스 갱신을 고려했는가.
- [ ] 테스트 가능한 단위로 로직을 분리했는가.
- [ ] 누락된 엣지케이스가 없는가.

### 5. 사용자 합의
- 완성된 플랜을 사용자에게 제시하고 **착수 전 동의**를 받는다.
- 규모가 크거나 대안이 갈리면 `EnterPlanMode`/`ExitPlanMode`로 승인 흐름을 쓴다.

플랜 합의가 끝나면 파이프라인 5단계(코드 작성)로 넘어간다.
